# InsureRAG 优化路线图

本文档整理 InsureRAG 当前可优化点，按重要程度排序。每项包括问题、优化方案、验收方式和面试表达，方便后续迭代和面试答辩。

## 当前项目定位

InsureRAG 当前是一个本地 RAG 原型项目，已完成：

- 真实保险 PDF 解析与文本清洗
- PDF 文本层质量预检
- 父子分块
- BGE-M3 Dense/Sparse 混合检索
- RRF 排名融合
- BGE-Reranker 重排序
- Qwen3-8B-AWQ 本地生成
- RAGAS 自动评测

但它还不是生产级系统。当前最值得优化的是检索质量、工程接口和评测实验体系。

## P0：父块去重与上下文压缩（已完成）

### 问题

当前 Milvus 中存储的是子块，一个较长父条款会切出多个子块。项目已在 Reranker 精排后按父块去重，避免多个子块对应同一 `parent_text` 时重复进入最终上下文。

影响：

- 重复父块占用 Top K 位置。
- 其他相关条款被挤出上下文。
- Prompt 变长，RAGAS 和生成模型更容易触发上下文长度限制。
- 指标中的 Context Precision 可能被重复上下文干扰。

### 优化方案

在 Reranker 精排后，按父块去重，再取 Top K。

当前短期可用：

```python
key = (source, parent_text)
```

更规范的方案：

```python
key = (source, parent_id)
```

这要求在 `stage2_build_db.py` 的 Milvus schema 中增加 `parent_id` 字段，并在 `stage3_search.py` 的 `output_fields` 中返回。

### 推荐实现顺序

1. 在 `stage2_build_db.py` 中为 collection 增加 `parent_id` 字段。
2. 插入数据时写入 `parent_id`。
3. 在 `stage3_search.py` 中召回 Top 20。
4. 对 Top 20 执行 Reranker。
5. 按 `(collection_name, document_id, parent_id)` 去重。
6. 去重后取 Top 3 或 Top 5。

### 验收方式

- Top K 中不再出现重复父条款。
- RAGAS 超长上下文报错减少或消失。
- Context Precision 不下降。
- 错误样例中相关条款覆盖率提升。

### 面试表达

> 父子分块会带来一个副作用：长父块的多个子块可能同时命中，导致完整父条款重复进入上下文。我会在 Reranker 后按 `(source, parent_id)` 做父块级去重，保证 Top K 覆盖更多不同条款，降低上下文冗余和 Token 消耗。

## P0：三个已知检索错误修复

### 问题 1：年龄限制问题召回错误

问题：

```text
投保康宁终身保险有年龄限制吗？
```

理想条款：

```text
第二条 投保范围
```

错误倾向：

```text
第二十条 年龄计算及年龄、性别错误处理
```

原因：

- 问题中“年龄限制”与第二十条“年龄错误处理”词面相似。
- Reranker 更偏向包含大量“年龄”相关表达的条款。

优化方向：

- 在评测问题中明确“投保范围”或“被保险人投保年龄”。
- 增加 query rewrite A/B，改写为：

```text
康宁终身保险投保范围 被保险人年龄上限 七十周岁以下
```

- 增加 parent dedup 后观察第二条是否能进入 Top K。

### 问题 2：保费交不起问题有歧义

原问题：

```text
保费交不起了，除了退保还有别的办法吗？
```

可能对应：

- 第七条：自动垫交
- 第八条：合同效力恢复
- 第九条：减额交清
- 第十五条：保单借款

这不是单纯检索错误，而是问题本身有多种合理答案。

优化方向：

- 若评测目标是第九条，应改成：

```text
保费交不起又不想退保，能不能减少保额把合同保留下来？
```

- 如果保留原问题，参考答案应覆盖多种可选方案。

### 问题 3：理赔时间限制问题有歧义

原问题：

```text
申请理赔有时间限制吗？
```

可能对应：

- 第十三条：出险后十日内通知
- 第十四条：请求给付保险金权利五年不行使而消灭
- 第十四条：达成给付协议后十日内赔付

优化方向：

- 若评测目标是请求权时效，应改成：

```text
被保险人或受益人请求保险公司给付保险金的权利，最晚几年内行使？
```

- 如果保留原问题，应先反问澄清，或回答多个时间限制。

### 验收方式

- 三个问题的正确父条款进入 Top 3。
- 答案正确性提升。
- 人工检查答案不再引用错误条款。

### 面试表达

> 我会把错误分为两类：真正的检索排序错误和问题歧义。年龄问题属于排序错误；保费交不起和理赔时间限制属于业务语义不唯一，需要改评测问题或支持澄清式回答。

## P1：检索实验对比表

### 问题

当前项目实现了混合检索和 Reranker，但缺少系统性的 ablation study，无法量化每个模块带来的收益。

### 实验组设计

建议至少做以下对比：

| 实验组 | 说明 |
| --- | --- |
| Dense only | 只使用 BGE-M3 Dense 向量召回 |
| Sparse only | 只使用 BGE-M3 Sparse 向量召回 |
| Dense + Sparse | 双路召回，RRF 融合 |
| Hybrid + Reranker | 混合召回后使用 BGE-Reranker 精排 |
| Hybrid + Reranker + Parent Dedup | 加入父块去重 |

### 建议指标

- Recall@3
- Recall@5
- MRR
- Context Precision
- Context Recall
- Answer Correctness
- 平均延迟
- P95 延迟

### 实现方式

新增脚本：

```text
stage7_retrieval_ablation.py
```

脚本逻辑：

1. 读取同一批评测问题。
2. 对每个检索策略返回 Top K。
3. 判断目标条款是否命中。
4. 输出 Markdown 或 CSV 表格。

### 面试表达

> 我不会只说用了混合检索和重排，而是会通过消融实验验证 Dense、Sparse、RRF、Reranker 和父块去重分别带来的收益。

## P2：Query Rewrite A/B 测试

### 当前状态

项目曾包含一个独立的 Query Rewrite / 意图识别实验脚本（`stage5_intent.py`），但它从未接入主问答与 RAGAS 评测链路，已在瘦身时移除。这里保留为优化方向，下面给出真正落地时应采用的设计（双路检索而非直接替换原问题）。

### 为什么不能直接启用

- 大模型改写可能改变原意。
- 保险问题中的时间、否定、免责条件很敏感。
- 会增加一次模型调用，带来额外延迟。
- 当前没有证明它能提升召回。

### 推荐方案

不要用改写问题替换原问题，而是双路检索：

```text
原问题检索
    +
改写问题检索
    |
合并去重
    |
Reranker
    |
父块去重
```

### 实验方式

对比：

- 原问题检索
- 改写问题检索
- 原问题 + 改写问题双路检索

指标：

- Recall@3
- Context Precision
- Context Recall
- Answer Correctness
- 平均延迟

### 面试表达

> Query Rewrite 不是一定正收益。我会通过 A/B 测试验证，并优先采用原问题加改写问题的双路检索，避免改写错误覆盖用户原意。

## P2：会话管理

### 问题

当前问答是单轮的，不支持上下文追问。

示例：

```text
用户：酒后驾驶赔吗？
用户：那无证驾驶呢？
```

第二个问题需要结合上一轮上下文。

### 优化方案

- 保存 session_id。
- 存储最近 N 轮对话。
- 对追问做 query condensation。
- 将省略问题改写为完整问题。

示例：

```text
那无证驾驶呢？
→ 无有效驾驶执照驾驶导致保险事故是否属于责任免除？
```

### 面试表达

> RAG 多轮对话的关键不是把历史全部塞进 Prompt，而是把追问改写为独立完整问题，再进入检索链路。

## P2：权限与安全

## P3：更严格的 PDF 质量门禁

### 当前状态

当前已在 `stage1_load_split.py` 中加入基础质量预检：抽取前 3 页文本，计算有效字符率和文本密度。若文本质量很差，会进一步检测 PDF 前几页是否包含图片对象；疑似图片型 PDF 时使用 RapidOCR 识别后再进入分块流程。

阈值策略：

- 有效字符率 >= 0.80：质量较好，正常入库。
- 0.60 <= 有效字符率 < 0.80：可入库，但打印警告，建议人工抽查。
- 有效字符率 < 0.60：高风险，跳过该 PDF，建议转 OCR 或人工清洗。
- 若高风险 PDF 的图片页占比 >= 0.50：判定为疑似图片型 PDF，触发 RapidOCR fallback。

### 后续优化方案

进一步可增加：

- 中文字符比例
- 乱码比例
- 条款编号连续性
- 重复短语比例
- 空页比例
- 分块数量异常
- 页眉页脚重复

### 面试表达

> 文档解析质量决定 RAG 上限。当前我已经在入库前加入质量门禁，用有效字符率和文本密度过滤明显异常的 PDF；如果进一步判断为图片型 PDF，则走 RapidOCR fallback。后续可以继续加入乱码率、空页率和条款编号连续性检查。

## 面试总结话术

> 这个项目已经完成 RAG 的核心闭环，但我不会把它包装成生产级系统。当前最优先的优化是父块去重和检索错误修复，其次是做检索消融实验、FastAPI 服务化和流式输出。如果要生产化，还需要多文档路由、权限隔离、日志监控、缓存、并发压测和更大规模评测集。
